# Manual Setup Guide (AWS Console)

Use this guide if you'd rather click through the AWS Console instead of
running `scripts/deploy.sh`. It mirrors exactly what the CloudFormation
template automates.

## 1. Enable CloudTrail

1. Open **CloudTrail** → **Trails** → **Create trail**.
2. Name: `threat-detection-trail`.
3. Choose **Create new S3 bucket** (enable default encryption, block all
   public access).
4. Enable **Log file validation** and **Apply trail to all regions**.
5. Leave management events enabled (Read + Write).
6. Create the trail.

## 2. Enable GuardDuty

1. Open **GuardDuty** → **Get Started** → **Enable GuardDuty**.
2. Leave the default finding frequency, or set it to **15 minutes** to
   match the CloudFormation default.
3. GuardDuty will automatically begin analyzing your CloudTrail
   management events, VPC Flow Logs, and DNS query logs — no additional
   wiring to CloudTrail is required.

## 3. Create the SNS Topic

1. Open **SNS** → **Topics** → **Create topic** → **Standard**.
2. Name: `threat-detection-alerts`.
3. After creation, go to **Create subscription**:
   - Protocol: `Email`, Endpoint: your email address.
   - Protocol: `SMS`, Endpoint: your phone number in E.164 format
     (e.g. `+15551234567`).
4. Confirm the email subscription via the link sent to your inbox.

## 4. Create the Lambda Function

1. Open **Lambda** → **Create function**.
2. Name: `guardduty-threat-processor`, Runtime: **Python 3.12**.
3. Under **Permissions**, create a new execution role, then attach an
   inline policy granting:
   - `sns:Publish` on the `threat-detection-alerts` topic ARN.
   - (Optional) `bedrock:InvokeModel` on
     `arn:aws:bedrock:<region>::foundation-model/*` if using AI enrichment.
   - See `iam/lambda-execution-role-policy.json` for the exact policy.
4. Upload `lambda/threat_processor.py` and `lambda/enrichment.py` (or zip
   both and upload via **Upload from → .zip file**).
5. Set the handler to `threat_processor.handler`.
6. Under **Configuration → Environment variables**, add:
   - `SNS_TOPIC_ARN` = the ARN of `threat-detection-alerts`.
   - `MIN_SEVERITY` = `4.0` (or your preferred threshold).
   - `USE_BEDROCK` = `false` (or `true` to enable AI enrichment).
   - `BEDROCK_MODEL_ID` = `anthropic.claude-3-5-sonnet-20241022-v2:0`
     (only needed if `USE_BEDROCK=true`).
7. Set **Timeout** to 30 seconds and **Memory** to 256 MB under
   **Configuration → General configuration**.

## 5. Wire Up EventBridge

1. Open **EventBridge** → **Rules** → **Create rule**.
2. Name: `guardduty-findings-to-lambda`.
3. Event bus: `default`. Rule type: **Rule with an event pattern**.
4. Event pattern (JSON edit mode):
   ```json
   {
     "source": ["aws.guardduty"],
     "detail-type": ["GuardDuty Finding"]
   }
   ```
5. Target: **Lambda function** → `guardduty-threat-processor`.
6. Create the rule. EventBridge automatically grants itself invoke
   permission on the target Lambda when created through the console; if
   you script this via CLI/CDK, remember to add an
   `AWS::Lambda::Permission` for `events.amazonaws.com` explicitly.

## 6. Test It

From the GuardDuty console: **Settings → Sample findings → Generate
sample findings**. Within a few minutes, you should receive an Email/SMS
alert generated by the Lambda function for every finding at or above your
`MIN_SEVERITY` threshold.

You can also tail the Lambda logs while testing:

```bash
aws logs tail /aws/lambda/guardduty-threat-processor --follow
```
